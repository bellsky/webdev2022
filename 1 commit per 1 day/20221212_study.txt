    12장. 침입 차단 시스템 Firewall

        12.2장 침입 차단 시스템(= Firewall) 특성

            Firewall 설계목표

            계획 및 구현 시 중요 요소
                적합한 접근 정책의 구체화.

            Firewall 역할과 범위

                1. 단 하나의 길목을 두어 취약한 서비스가 네트워크에서 출입하는 것을 금지하고,
                다양한 종류의 IP 스푸핑과 라우팅 공격으로부터 보호.
                2. 보안관련 사건의 위치 추적, 감사, 경고 구현
                3. 보안과 연관하지 않은 여러 인터넷 기능을 수행할 수 있는 편리한 플랫폼
                4. IPsec으로 활용할 수 있다
                    (터널 모드를 이용한 가상 사설망 구성 가능)
            Firewall 제약 조건

                -자신을 우회하는 공격으로부터는 보호 불가 ex-전화 연결 기능을 통한 접속
                -불평분자 직원, 외부 공격자와 협력하는 직원 등 내부 위협에 취약.
                -안전 조치가 미흡한 무선 LAN은 조직 외부로부터 접근 가능
                -랩톱, PDA, 휴대용 저장장치는 회사 네트워크 밖에서 감염될 가능성이 있음

        12.3 침입차단시스템 유형
            -방화벽
            -Firewall 유형
                4가지
            

            패킷필터 Firewall의 약점

                1. 패킷-필터링 Firewall
                    : 들어오는 IP패킷에 규칙모음을 적용. 
                    양쪽 방향에서 들어오는 패킷 모두를 필터링한다.
                    필터링 규칙은 네트워크 패킷에 포함된 정보(발신지와 목적지의 IP주소,
                        목적지 전송 계층 주소. 그리고 IP프로토콜 필드, 인터페이스)에 기초함.

                    -패킷 필터 = 규칙의 집합 set. IP나 TCP 헤더 안의 필드와 
                    이 set이 일치하는 지 검사함.
                    SMTP, HTTP에서 쓰인다.

                2. 스테이트풀 검사 Firewall
                    : 전통적 패킷 필터. 패킷 하나하나마다 필터링함.
                    모든 포트번호를 가지고 들어오는 패킷을 통과시킨다.
                    패킷 필터링과 비슷하지만, TCP 연결에 관한 정보를 기록한다는 차이가 있다.
                    또한 제한적으로 응용 데이터를 ㅈ사하여 잘 알려진 프로토콜을 탐색한다.

                3. 응용-레벨 게이트웨이 Firewall
                    : 프록시 서버라고도 한다.
                    응용-레벨 트래픽 중개자 역할을 한다.
                    패킷 필터보다 더 안전하다.
                    응용 레벨에서 들어오는 트래픽을 기록하고 감사하기 용이하다.
                    각 연결마다 부가적 처리 오버헤드가 발생한다는 단점이 있다.

                4. 회선-레벨 게이트웨이 Firewall
                    : 종단 대 종단 TCP 연결을 허용하지 않는다.
                    두개의 TCP 연결(자신과 내부 호스트의 사용자 사이,
                        자신과 외부 호스트의 tCP 사용자 사이)을 설정한다.
                    연결 설정 이후 게이트웨이는 내용검사를 하지 않고 한 연결에서 다른 연결로
                    TCP 단편을 중개해서 전달한다.

                    시스템 관리자가 내부 사용자를 신뢰하는 상황에 사용된다.

                    구현 예시 : SOCKS

        12.4 침입차단 시스템 배치

            -일반적으로 운영체제를 쓰는 독립된 머신에 배치됨.
            -라우터나 LAN 스위치 안에서 소프트웨어 모듈 형태로 구현가능하다.


            배스천 호스트 Bastion host
                : Firewall 관리자에 의해 네트워크 보안에서의 핵심으로 인식되는 시스템.
                보통 응용-레벨 혹은 회선-레벨 게이트웨이용 플랫폼으로 이용된다.

                -배스천 호스트의 일반적 특성
                    -하드웨어 플랫폼.
                    -필수 서비스만 설치
                    -프록시 서비스에 사용자 접근허용을 하기 전 추가적으로 인증을 요구한다.
                    -각 프록시는 표준 명령 모음 중 일부만 지원.
                    -각 프록시는 특정 호스트 시스템에게만 접근을 허용.
                    -각 프록시는 감사 정보를 유지.
                    -각 프록시 모듈은 네트워크 보안용으로 설계된,
                        아주 작은 소프트웨어 패킷임.
                    ...

            호스트 기반 Firewall
                : 개별 호스트를 안전하게 하는 소프트웨어 모듈
                    (많은 운영체제에서 사용 가능, 추가 패키지로 제공함.)
                전통적 독립 Firewall처럼 패킷을 필터링하고 패킷 흐름을 제어.
                설치 장소는 서버임.

                장점 : 필터링 규칙을 호스트 맞춤 가능.
                        토폴로지에 무관하게 보호 가능.
                        독립적 침입차단 시스템과 병합 사용시,
                            호스트-기반 침입차단 시스템은 추가적 보안 계층 제공 가능

            개인 Persnal Firewall
                -주요 역할 : 허가받지 않은 컴퓨터로의 원격접속을 차단하는 것.

    12장. 예외처리
        
        기존에는 인터럽트로 넘겨서 프로그램을 종료해야 했던 문제에서
            예외처리를 통해 프로그램이 계속 실행될 수 있도록 하는 것이 목적임.


        설계쟁점


        각 언어별

            -C++  : try  catch

                사용자 정의 예외만 존재한다.
                시승템 예외처리는 불가능하다. ex) 디바이드바이제로

            -Java  : 
                C++보다 개선된 점.
                발생될 수 있는 객체외 프로그램에 있는 모든 다른 객체를 구분함.
                사용자 예외 처리기에서 처리될 수 있는 여러가지 시스템 예외의 묵시적 발생.
                finally절의 추가로 복합문이 종료되는 것에 상관없이 청소 작업의 발생 허용
                header로부터 발생할 수 있지만 처리하지 않은 예외를 알 수 있다.
